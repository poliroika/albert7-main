## 4.1 Зачем отдельное дерево `.umbrella/`

Репозиторий сознательно делит **исходники** (то, что коммитят) и **операционный след** (логи, кэши состояния Ouroboros, web-store, файлы остановки запусков). Корнем для последнего служит `.umbrella/` в корне checkout. Точный состав может эволюционировать вместе с кодом; ниже перечислены устойчивые категории.

## 4.2 Типичные подкаталоги и файлы

| Путь (примерный смысл) | Назначение |
|------------------------|------------|
| `.umbrella/app_ouroboros.log` | Лог entrypoint `app_ouroboros` (stdout дублируется в файл) |
| `.umbrella/launcher/` | Служебные маркеры launcher (в т.ч. stop-request для кооперации с UI) |
| `.umbrella/ouroboros_drive/` | Диск состояния и результатов Ouroboros: state, task_results, verification JSON |
| `.umbrella/web/` | Persisted настройки и данные UI bridge (`util.store_path` и аналоги) |
| `.umbrella/meta_harness/` | Store экспериментов Meta-Harness (кандидаты, снимки, решения promotion) |

Не каждый запуск создаёт все ветки; отсутствие каталога часто означает просто «ещё не трогали эту подсистему».

## 4.3 Что оператору имеет смысл открывать при инциденте

1. **Последние строки лога приложения** — быстрый narrative.
2. **`task_results` и `*.verification.json`** — формальный отчёт verification и статус итерации (удобно стыковать с Web «Runs»).
3. **Stop-request файлы** — если UI или оператор запросили остановку, а процесс ведёт себя странно.

## 4.4 Гигиена

- Перед долгими прогонами имеет смысл проверять место на диске и ротацию логов.
- Часть артефактов может содержать чувствительные фрагменты (промпты, пути); политика коммита — не включать `.umbrella/` в публичные ветки без фильтрации.
