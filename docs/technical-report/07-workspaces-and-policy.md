## 7.1 Что считается workspace

Workspace — каталог под `workspaces/<id>/`, содержащий всё необходимое, чтобы **поднять прикладную систему на GMAS** и проверить её теми же средствами, что использует оператор: тесты, скрипты, локальные сервисы. Идентификатором в большинстве API выступает **имя последней компоненты пути** (`<id>`).

## 7.2 Seed и task-instance

- **Seed** — эталон. Его не следует превращать в полигон для одноразовых черновиков; иначе теряется смысл шаблона.
- **Task-instance** — рабочая копия под конкретную миссию. Автоматические правки Ouroboros по умолчанию направляются сюда (`--require-instance` в CLI; противоположный полюс — `--allow-seed-writes`).

Подробные сценарии создания и эволюции: [../creating-workspaces.md](../creating-workspaces.md) и [../workspaces.md](../workspaces.md).

## 7.3 Ключевые файлы контракта

| Файл | Роль |
|------|------|
| `workspace.toml` | Метаданные, секции инструментов, **`[verification]`** — декларация проверок |
| `TASK_MAIN.md` | Дефолтное текстовое задание для непрерывного улучшения и для CLI без `--task` |

Отсутствие или некорректность секции verification приводит к авто-детекту или к repairable-ошибке спеки — см. [09-verification.md](09-verification.md).

## 7.4 Политика «не ломать gmas»

Изменения в `gmas/` проходят отдельную планку доверия. Workspace должен оставаться **возможным к запуску без Ouroboros** в разумных пределах — это страховка от привязки всей компетенции к менеджеру.

## 7.5 Взаимодействие с registry и UI

Список workspace’ов в Web bridge строится из registry-слоя. Если каталог существует, но не виден в UI, проверяют:

- соответствие ожидаемой структуры;
- ошибки при чтении `workspace.toml`;
- фильтры в API (редко).
