# TASK: Multi-Agent Debate Graph (gMAS)

## Goal

Спроектировать и реализовать **мультиагентную систему аргументации с синтезом** на **gMAS** (`gmas/`): пользователь задаёт **один открытый вопрос или тезис**; система строит **разветвлённый граф** (не только линейный пайплайн), согласует выходы и выдаёт **сбалансированный итог** с явными оговорками.

### Роли (минимум 4)

1. **Framer** — уточняет формулировку вопроса, выделяет критерии «что считать успехом ответа», ограничения (например «только публичные данные», «без медицинских советов»).
2. **Advocate** — строит сильную линию **«за»** тезис (или за одну из сторон), со структурой: тезисы → аргументы → допущения.
3. **Skeptic** — строит сильную линию **«против»** (или за противоположную сторону), с тем же форматом; **не** дублирует Advocate дословно, а атакует слабые места и альтернативные объяснения.
4. **Synthesizer** — объединяет выводы Framer + Advocate + Skeptic в **единый документ**: где согласие, где конфликт, что остаётся неизвестным, практическая рекомендация «если нужно решение сегодня».
5. **Critic** (пятая роль) — проверяет синтез на **баланс** (нет ли перекоса), на **галлюцинации** относительно входа и на логические дыры; при необходимости одна итерация **уточнения только текста синтеза** (без повторного полного дерева Advocate/Skeptic), лимит **1** круг.

### Граф (концепция)

- Линейный старт: **Framer** → затем **параллельно или последовательно** (на усмотрение реализации, но в README зафиксировать) **Advocate** и **Skeptic**, оба получают выход Framer.
- **Synthesizer** принимает все три артефакта.
- **Critic** завершает или инициирует одну правку Synthesizer.

Использовать публичный API gMAS (`GraphBuilder`, раннер, роли, рёбра — по актуальной документации в `gmas/docs`).

## Functional requirements

1. Точка входа: **CLI** (аргумент: вопрос/тезис строкой или `--file path.txt`).
2. Выход: **stdout** и/или файл в `output/` (например markdown с секциями pro / con / synthesis / critic).
3. Конфигурация LLM только через **env**; **`.env.example`** без секретов.
4. Обработка пустого ввода, ошибок узла, таймаута — понятные сообщения и код выхода.
5. **pytest**: unit на парсинг/валидацию входа; smoke с **моком** LLM или раннера (без обязательного реального API в CI).

## Non-functional requirements

- Зависимости зафиксированы после появления кода (`pyproject.toml` или `requirements.txt`).
