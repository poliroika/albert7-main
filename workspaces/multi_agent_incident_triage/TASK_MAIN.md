# TASK: Multi-Agent Incident Triage (gMAS)

## Goal

Реализовать **мультиагентный пайплайн разбора входящего инцидента** (текст тикета, алёрта или постмортем-черновика) на **gMAS** из монорепозитория (`gmas/`): минимум **четыре роли**, явный граф и передача **структурированных артефактов** между узлами (JSON или чёткие секции markdown), один сценарий запуска (CLI / `python -m …`), конфигурация LLM только через **переменные окружения**, **`.env.example`**, **pytest** с моком раннера или LLM.

**Сюжет домена (зафиксировать в README при реализации):**

1. **Triage** — классифицирует серьёзность, область (security / infra / data / product), извлекает сущности (сервисы, версии, окружения).
2. **Investigator** — формирует список гипотез и вопросов «что проверить», без выдуманных фактов; опирается только на вход и вывод Triage.
3. **Responder** — собирает черновик ответа заинтересованным сторонам (кратко: что случилось, что известно, следующие шаги, риски).
4. **Critic** — проверяет черновик на противоречия с Triage/Investigator, пустые обещания и отсутствующие явные unknowns; либо одобряет, либо возвращает замечания в явном формате для одного цикла доработки Responder (не более **одного** круга Critic→Responder, чтобы граф оставался управляемым).

Итог — **рабочий код в этом каталоге** + README с диаграммой графа и командами запуска/тестов.

## Functional requirements

1. Граф на gMAS: узлы, рёбра, контракт данных между Triage → Investigator → Responder и ветка **Critic → Responder** (один раз) при несогласии.
2. Вход: строка или путь к файлу с текстом инцидента (аргумент CLI).
3. Выход: финальный артефакт (например `stdout` + опционально запись в `output/`): сводка + черновик ответа + блок «open questions».
4. Ошибки: пустой ввод, сбой узла, таймаут — понятное сообщение и ненулевой exit code при фатале.
